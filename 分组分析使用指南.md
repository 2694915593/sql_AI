# SQL分组分析功能使用指南

## 功能概述

分组分析功能实现了以下核心能力：
- **按文件名分组**：自动将同一文件名（FILENAME）的SQL语句进行分组
- **结果组合**：将同一组的多个SQL分析结果合并为统一的AI分析报告
- **智能存储**：将组合结果存储到 `AM_COMMIT_SHELL_INFO` 表的 `AI_ANALYSE_RESULT` 字段
- **自动关联**：依据 `PROJECT_ID`、`DEFAULT_VERSION`、`FILENAME` 等字段进行精准关联

## 系统要求

### 1. 数据库表结构

#### 源表：AM_SQLLINE_INFO（存储待分析SQL）
必须包含以下关键字段：
```sql
CREATE TABLE AM_SQLLINE_INFO (
    ID INT PRIMARY KEY,
    SQL_TEXT TEXT,           -- SQL语句内容
    FILENAME VARCHAR(255),    -- 文件名
    PROJECT_ID VARCHAR(100),  -- 项目ID
    DEFAULT_VERSION VARCHAR(100), -- 默认投产分支
    FILE_PATH VARCHAR(500),   -- 文件路径
    SYSTEM_ID VARCHAR(100),   -- 系统ID
    AUTHOR VARCHAR(100),      -- 作者
    OPERATE_TYPE VARCHAR(50), -- 操作类型
    STATUS VARCHAR(20),       -- 分析状态
    ANALYSIS_RESULT JSON,     -- 单个SQL分析结果
    CREATE_TIME DATETIME,
    UPDATE_TIME DATETIME
);
```

#### 目标表：AM_COMMIT_SHELL_INFO（存储分组结果）
必须包含以下关键字段：
```sql
CREATE TABLE AM_COMMIT_SHELL_INFO (
    ID INT PRIMARY KEY AUTO_INCREMENT,
    PROJECT_ID VARCHAR(100),      -- 项目ID
    DEFAULT_VERSION VARCHAR(100), -- 默认投产分支
    CLASSPATH VARCHAR(500),       -- 类路径/文件路径
    FILENAME VARCHAR(255),        -- 文件名
    AI_ANALYSE_RESULT JSON,       -- AI分析结果（组合后的JSON）
    CREATE_TIME DATETIME,
    UPDATE_TIME DATETIME,
    INDEX idx_project_file (PROJECT_ID, DEFAULT_VERSION, FILENAME)
);
```

## 使用方法

### 1. 准备工作

1. **确认表结构**：确保 `AM_SQLLINE_INFO` 和 `AM_COMMIT_SHELL_INFO` 表存在且结构正确
2. **准备数据**：
   - 在 `AM_SQLLINE_INFO` 表中插入待分析的SQL记录，确保 `FILENAME`、`PROJECT_ID`、`DEFAULT_VERSION` 字段有值
   - 在 `AM_COMMIT_SHELL_INFO` 表中预存分支和文件信息（用于识别本次投产分支的文件）
3. **配置文件**：修改 `config.ini` 中的数据库连接信息和AI API配置

### 2. 运行分组分析

#### 方式一：分支模式（推荐，按新需求实现）
这是根据新需求实现的分组逻辑，首先通过 `AM_COMMIT_SHELL_INFO` 表获取本次投产分支的所有文件，然后按文件名和分支分组分析：

```bash
# 进入项目目录
cd e:\Code\sqlAnalize

# 运行分支分析（指定分支名称）
python sql_ai_analyzer/main.py --mode branch --branch-name v1.0.0

# 指定批处理数量
python sql_ai_analyzer/main.py --mode branch --branch-name v1.0.0 --batch-size 20

# 指定配置文件路径
python sql_ai_analyzer/main.py --mode branch --branch-name v1.0.0 --config sql_ai_analyzer/config/config.ini
```

#### 方式二：批处理模式（传统方式）
这是之前的分组逻辑，按文件名分组（兼容旧版本）：

```bash
# 运行批处理分析（默认处理10条SQL）
python sql_ai_analyzer/main.py --mode batch

# 指定批处理数量
python sql_ai_analyzer/main.py --mode batch --batch-size 20
```

#### 方式三：单个SQL分析（不触发分组）
如果只需要分析单个SQL，不会触发分组处理：

```bash
python sql_ai_analyzer/main.py --mode single --sql-id 123
```

#### 方式四：持续运行模式
用于监控和持续处理新SQL：

```bash
# 每5分钟检查一次新SQL并自动分组处理
python sql_ai_analyzer/main.py --mode continuous --interval 300
```

### 3. 新的分组处理流程（分支模式）

当运行分支模式时，系统会执行以下流程：

```
1. 从AM_COMMIT_SHELL_INFO表获取指定分支的所有文件
   ↓
2. 对每个文件，到AM_SQLLINE_INFO表查找相同文件名和分支的SQL
   ↓
3. 按文件名和分支进行分组
   ↓
4. 逐条分析组内每个SQL
   ↓
5. 收集分析结果
   ↓
6. 组合同一文件的分组结果
   ↓
7. 存储到AM_COMMIT_SHELL_INFO表的AI_ANALYSE_RESULT字段
   ↓
8. 返回处理摘要
```

### 4. 传统分组处理流程（批处理模式）

当运行批处理模式时，系统会执行以下流程：

```
1. 提取待分析SQL
   ↓
2. 逐条分析每个SQL
   ↓
3. 收集分析结果
   ↓
4. 按文件名分组
   ↓
5. 组合分析结果
   ↓
6. 存储到AM_COMMIT_SHELL_INFO表
   ↓
7. 返回处理摘要
```

## 新分组逻辑详解（分支模式）

### 数据流向
```
AM_COMMIT_SHELL_INFO表（分支文件信息）
        ↓
获取指定分支的所有文件（按DEFAULT_VERSION字段）
        ↓
AM_SQLLINE_INFO表（待分析SQL）
        ↓
查找相同文件名和分支的SQL（FILENAME + DEFAULT_VERSION）
        ↓
按文件名和分支进行分组
        ↓
分析并组合结果
        ↓
存储到AM_COMMIT_SHELL_INFO表（AI_ANALYSE_RESULT字段）
```

### 分组键生成（新逻辑）
分组基于以下三个字段的组合：
- **FILENAME**：文件名（来自AM_COMMIT_SHELL_INFO表）
- **PROJECT_ID**：项目ID（来自AM_COMMIT_SHELL_INFO表）  
- **DEFAULT_VERSION**：默认投产分支（来自AM_COMMIT_SHELL_INFO表）

生成的分组键格式：`{FILENAME}_{PROJECT_ID}_{DEFAULT_VERSION}`

### 查询逻辑
1. **获取分支文件**：
   ```sql
   SELECT DISTINCT FILENAME, PROJECT_ID, CLASSPATH, DEFAULT_VERSION
   FROM AM_COMMIT_SHELL_INFO 
   WHERE DEFAULT_VERSION = '分支名称'
   ```

2. **获取文件SQL**：
   ```sql
   SELECT ID, SQL_TEXT, FILENAME, PROJECT_ID, DEFAULT_VERSION
   FROM AM_SQLLINE_INFO 
   WHERE FILENAME = '文件名' 
     AND DEFAULT_VERSION = '分支名称'
     AND (STATUS IS NULL OR STATUS = 'pending' OR STATUS = 'new')
   ```

### 结果组合规则
1. **文件信息**：保留文件的基本信息（文件名、项目ID、分支、路径）
2. **SQL列表**：保留每个SQL的ID、预览、类型和评分
3. **建议合并**：去重后按SQL分组展示建议
4. **风险评估**：汇总所有SQL的风险问题，按高/中/低风险分类统计
5. **综合评分**：计算组内所有SQL的平均分
6. **详细数据**：保留每个SQL的完整分析数据用于后续查询

## 传统分组逻辑详解（批处理模式）

### 分组键生成（传统逻辑）
分组基于以下三个字段的组合：
- **FILENAME**：文件名（来自AM_SQLLINE_INFO表）
- **PROJECT_ID**：项目ID（来自AM_SQLLINE_INFO表）  
- **DEFAULT_VERSION**：默认投产分支（来自AM_SQLLINE_INFO表）

生成的分组键格式：`{FILENAME}_{PROJECT_ID}_{DEFAULT_VERSION}`

### 两种模式对比

| 特性 | 分支模式（新） | 批处理模式（传统） |
|------|---------------|------------------|
| **数据源** | AM_COMMIT_SHELL_INFO表为主 | AM_SQLLINE_INFO表为主 |
| **分组依据** | 先查分支文件，再找对应SQL | 直接按SQL文件名分组 |
| **适用场景** | 按投产分支批量分析 | 通用批量分析 |
| **数据依赖** | 需要AM_COMMIT_SHELL_INFO表有预存数据 | 只需要AM_SQLLINE_INFO表 |
| **分组精准度** | 高（确保分支一致性） | 中（可能存在跨分支同名文件） |
| **运行命令** | `--mode branch --branch-name xxx` | `--mode batch` |

## 故障排除

### 常见问题

#### 1. 分组处理未触发
**现象**：批处理完成后没有分组处理日志
**原因**：没有成功的分析结果或配置文件问题
**解决**：
- 检查是否有SQL分析成功（`success_count > 0`）
- 确认 `AM_SQLLINE_INFO` 表中的 `FILENAME` 字段不为空
- 检查数据库连接是否正常

#### 2. 分组键生成失败
**现象**：`group_key` 显示为 `unknown_{sql_id}`
**原因**：`FILENAME` 字段为空
**解决**：确保 `AM_SQLLINE_INFO` 表中的 `FILENAME` 字段有值

#### 3. 存储到AM_COMMIT_SHELL_INFO失败
**现象**：分组处理显示 `failed` 状态
**原因**：目标表不存在或权限问题
**解决**：
- 确认 `AM_COMMIT_SHELL_INFO` 表存在且结构正确
- 检查数据库用户是否有INSERT/UPDATE权限
- 查看详细错误日志

#### 4. AI_ANALYSE_RESULT字段存储问题
**现象**：JSON数据存储失败
**原因**：JSON格式错误或字段类型不匹配
**解决**：
- 确认 `AI_ANALYSE_RESULT` 字段类型为JSON或TEXT
- 检查生成的JSON数据是否有效（可使用 `JSON_VALID()` 函数验证）
